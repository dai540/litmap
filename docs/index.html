Pipeline at a glance

If you only read one section before opening the tutorials, read this one. The package always moves through the same conceptual stages: define a corpus, prepare paper text, build embeddings, cluster in analysis space, and only then project into a separate 2D display space.

  1. Sources and corpus prep: collect records from PubMed, OpenAlex, Semantic Scholar, or local files, then normalize them into one paper table.
  2. Embedding stage: convert title and abstract text into vectors that preserve thematic proximity between papers.
  3. Analysis space: normalize and reduce the embedding space for clustering; this is where HDBSCAN decides cluster membership.
  4. Display space: build a separate 2D PCA only for plotting, so the map stays readable without changing cluster logic.
  5. Artifacts: save labels, summaries, coordinates, PCA tables, and an HTML map so the run can be inspected later.

The most important design choice is the split between analysis space and display space. Cluster count comes from the analysis pipeline, not from the 2D picture.

A good package README should not carry the entire documentation burden. It should hand off to a richer HTML documentation site with multiple articles, just as `pkgdown` does in the R ecosystem.
